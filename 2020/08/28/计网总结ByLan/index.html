1. 计算机网络和因特网

​ 在对计算机网络进行深度学习之前，我们需要对网络总体有一个了解，掌握基本概念和术语，在这一章节中，我们以因特网为例来进行讲解。

1.1 什么是因特网

​ 当然，用因特网来进行讲解的前提是了解因特网（开始套娃），因特网是什么呢？我们这里分别用两种描述方式来对因特网进行描述

1.1.1. 根据硬件和软件的构成进行具体描述

硬件组成

终端设备：包括手机，PC，网络电视、服务器等等

通信链路：光纤，同轴电缆，无线电，卫星，双绞线

通信设备：路由器，交换机（二级交换机，三级交换机），中继器

软件组成：

互联网服务供应商：ISP（Internet Service Provider），比如我们熟知的中国电信，联通等等

协议：因特网的主要协议统称为TCP/IP。协议定义了两个或多个通信实体之间交换的报文格式和次序，以及报文发送/或接收一条报文或其他事件所采取的动作（因特网中所有的通信活动都受到协议的控制）

协议三要素：

消息语法：即数据与控制信息的结构或格式(IP 头部、TCP 头部的字段含义)——结构

消息语义：即需要发出何种控制信息，完成何种动作以及做出何种响应——内容

规则/同步：即事件实现顺序的详细说明——时序

互联网标准：RFC，IETF

1.1.2. 根据提供服务的角度来描述

分布式应用程序仅仅运行在终端系统上，不运行在网络核心的分组交换机中，因特网就是为应用程序提供服务的基础设施。

（这里还涉及一个概念——分布式应用程序：在端系统上运行，彼此可以通信。实现因特网的各种应用，如电子邮件，Web应用，即时讯息，分布式游戏等等）

现在我们知道了什么是因特网，就是一张将通信设备，通信链路，交换设别都连起来的“网”。范围很大，所以因特网又分为网络边缘和网络核心。

网络边缘包括：外围部件，主机（客户，服务器），网络应用（别遗漏了）

网络核心包括：路由器，通信链路，网络的网络

网络划分

在下面的两小节我们就来依次介绍上述两部分。

1.2 网络边缘

现在我们已经知道网络边缘包括外围部件，主机（客户、服务器），网络应用，这里又引入两个新的概念：

接入网：将端系统连接到其边缘路由器的物理链路。是用户连接到网络的基础设施。

边缘路由器：端系统到任何其他远程端系统的路径上的第一台路由器。

1.2.1 接入网

那么网络边缘如何连接到网络核心呢？也就是说如何将终端系统连接到边缘路由器呢？这也就是接入网的三种方式：

  1. 家庭接入：DSL，电缆，FTTH（光纤到户技术），拨号和卫星

    • DSL（Digital subscriber line）

      在DSL方式中，为了能够同时上网和接听电话采用了一种频分复用技术（FDM），这种技术可以同时传输语音信号和数据信号，其中数据信号的传输又可以用时提供上行和下行数据传输。

    • HFC混合光纤同轴电缆

  2. 企业接入：以太网（Ethernet），无线以太网（Wi-Fi） ——用于无线移动设备的接入

    无线用户通过基站（无线接入点）连接

    基站与有线的因特网连接，为无线用户提供服务，

    通过基站互相通信，覆盖范围约几十米

  1. 广域无线接入（移动接入/局域网接入）：3G，LTE, 4G , 5G

    基站由电信提供商管理，比如中国移动，联通等，覆盖范围数万米

    （LTE技术是3G的演进，是3G到4G的过渡）

1.2.2 物理媒体

  • 概念：将网络中不同端系统互相连接起来的物理线路，进行数据传输，通过传播电磁波或光脉冲来发送比特流，即进行数据传输。

    • 引导性媒体：电波沿着固体媒体传播。如双绞线，同轴电缆，光缆
    • 非引导性媒体：电波在空气或外层空间中传播，如无线电
      • 陆地无线电
        • 广域网的接入：3G/4G
        • 局域网无线接入：Wi-Fi
      • 卫星无线电（用在信号无法覆盖的地方）

    对每种物理媒体的描述这里就不详细赘述了，可以自行了解。

1.3 网络核心

概念：连接端系统的分组交换机和链路形成的网状网络

清楚网络核心和网络边缘之后，网络构建形式有哪些呢，也就是说数据是如何通过网络传送的呢？端到端的数据传输有两种基本技术：分组交换和电路交换。

  1. 分组交换（如因特网）

    • 概念：主机将应用层消息(messages)分割为若干分组(packets)，每个分组沿着从源主机到目的主机的传输路径通过每个路由器(逐跳)转发，每个分组在每条链路传输时，以链路的最大传输速率传输。每个路由器采用存储转发机制，即将每个分组完整接收后，根据其 IP 头部包含的目的 IP 地址进行转发，最后在目的地合并。
    • 两个关键的网络核心功能：
      • 路由：确定分组从源-目的主机的路径，通俗来说就是找“路径”的算法
      • 转发：将分组从路由器的输入端口转到适当的路由器的输出端口，通俗来说就是找输出端口

  2. 电路交换（如打电话）

    • 概念：数据的发送方和接收方之间需要建立一条物理的电路，发送方可以以恒定的速率向接收方发送数据，整条电路被临时占用，直到数据发送结束。通信双方必须先建立一个专用的连接（电路），一直维持，直到通信结束。对连续传送大量数据效率高。

    • 使用多路复用技术

      所谓多路复用，就是在一条传输链路上同时建立多条连接的电路，分别传输数据。

      • 频分多路复用FDM（frequency-division multiplexing）

        按频率划分为若干个频段，每个频段专用一个连接。

      • 时分多路复用TDM（time-division multiplexing）

        时间划分为固定区间的帧，每帧再划分为固定数量的时隙，每一个时隙专用于一个连接，用于传输数据。

资源预留 发送数据
分组交换 不需要资源预留 发送方以恒定速率向接收方发送数据
电路交换 预留端到端资源：预留端系统之间
通信路径上所需要的资源，建立连接		 按需使用资源，可能需要排队等待，同时有其他分组发送
  1. 报文交换（如电报网络）
    • 概念：整个报文先传送到相邻结点，全部存储下俩后，再转发到下一个结点

三种交换技术的对比：

电路交换：整个报文的比特流连续地从源点直达重点，好像在一个管道中传送

报文交换：整个报文先传送到相邻接地那，全部存储下来之后，再转发到下一个结点

分组交换：单个分组（只是一部分报文）传送到相邻接地那，存储下来之后，再转发到下一个结点

1.4 四种分组时延

在因特网中使用的是分组交换，这也是我们要讲的一种重点数据传输方式，上一小节我们了解了分组交换的概念和基本传输过程，接下来我们深入剖析分组交换中的一些概念。

因为分组交换中，分组在每个路由器中先存储，再转发。因此分组可能会出现排队现象，产生时延，甚至丢包。

  • 时延

    缓存中排队，等待交换

  • 丢包

    分组到达链路的速率超过输出链路的能力，就可能会出现丢包的现象。链路的缓冲区的容量是有限的，分组到达一个排满的队列时会产生丢包。

  • 吞吐量（Throughput）

    单位时间内整个网络传输数据的速率或分组数。这里还涉及瓶颈链路（bottlenneck link）—— min(R1,R2)，就是说端到端吞吐量取决于瓶颈链路。

  1. 节点处理时延dproc
    • 检查错误位,检测比特错误
    • 选择输出链路
    • 高速路由器处理延迟–微秒级别
  2. 排队时延dqueue
    • 等待被发送到输出链路上的时间
    • 取决于路由器的拥塞程度
  3. 传输时延dtrans
    • 传输就是输出，就是从节点输出到通信链路所花的时间
    • L=分组长度 (bits)，R=链路带宽 (bps)（也可称为信道带宽，也就是数据率）
    • 传输时延 =数据帧长度/链路带宽个 = L/R
  4. 传播时延dprop(propagation)
    • 传播就是在距离上传播花费的时间
    • d = 物理链路的长度，s = 介质的信号传播速度
    • 传播延迟 = 物理链路的长度/在信道上的传播速率 = d/s

端到端时延：就是上述四个时延的总和

1.5 协议层次及其服务模型

经过上面的学习，相信大家对因特网的一些基本概念，和数据传输方式都有了基本的了解。因特网的协议统称为TCP/IP协议，这是一个庞大的协议集合，我们学习要将其细分出来学习。因此在这一小节，我将介绍对协议层次及其服务模型的分层以及为什么要分层来进行讲解。

  • 网络协议为什么要分层？

    • 使复杂系统简化：将一个大而复杂的系统划分为若干个明确、特定的部分，分别研究讨论
    • 易于维护、系统的更新：某层功能变化，不影响其余部分（功能独立）

    通俗来说就是，协议就是写在纸上的需求，需求需要软件实现，软件大了都要分层，干脆写协议的时候就告诉软件该怎么分层。

    网络协议就是个软件需求说明书，把需求分分层很正常。一个网络包在互联网中跑，一定是电信号或者光信号跑在具体的硬件设备上。

  • 服务：各层通过其SAP向上一层提供的一组功能

    SAP就是一个层次系统的上下层之间进行通信的接口，N层的SAP就是N+1层可以访问N层服务的地方

分层协议结构

  • TCP/IP协议栈，五层模型及对应的数据封装：从高到低

    如果是四层模型的话，不含物理层，包含在链路层里

    • 主机实现所有五个层次
    • 路由器实现底部三个层次
    • 链路交换机只实现底部两个层次

  1. Application（应用层）—— 报文 message

    • 提供各种应用，传输应用层报文
    • 协议：FTP，SMTP，HTTP，DNS，DHCP（动态主机配置协议）

  2. Transport （传输层）—— 报文段 segement

    • 完成进程-进程的数据传输，
    • 协议：TCP，UDP

  3. Network（网络层）—— 数据报 datagram

    • 完成主机-主机的数据传输，与link相比只有一段
    • 协议：IP，RIP，OSPF，BGP，ICMP，ARP

  4. Link（链路层）—— 帧 frame

    • 相邻网络节点间的数据帧传送，实现单条链路的数据传输
    • 802.11，802.1，802.3

  5. Physical（物理层）—— 比特流 bits

    • 实现单条链路上比特流的编码和解码，比特层次上的传输（物理介质上的比特传送）

分层后数据传递的物理过程：发送方从上层到下层逐渐添加头部封装数据，接收方从下层往上层逐渐解封去除头部传递数据。

如果是OSI七层模型，还包含

  • 表示层:加解密、数据压缩、格式转换
  • 会话层:对数据传输进行管理，包括数据交换的定界、同步，建立检查点等。

但现在其实OSI模型已经淘汰了，仅做了解即可。我们只需要记住TCP/IP的五层模型即可。

具体分层已经完成，基本概念相信大家也已经明白。接下来我们就逐层开始讲解啦！！！冲冲冲！！

2. 应用层

（HTTP，FTP，SMTP，DNS）

  • 应用层是体系结构中最高层。直接为用户的应用程序（如电子邮件，文件传输和终端仿真）提供服务，网络应用程序都部署在终端系统上，基于网络通信，不涉及网络核心中的设备。

  • 在因特网中的应用层协议也很多，支持万维网的HTTP协议，支持电子邮件的SMTP协议，支持文件传送的FTP，DNS，POP3等

接下来我们就来依次讲解这些应用层协议如何在应用层中应用，首先了解一下应用层的协议原理吧！

2.1 应用层协议原理

在应用层运行的应用程序有以下两种体系结构：

2.1.1 网络应用程序体系结构

  • client-server (C/S,B/S)架构

    客户端首先发起连接请求，如HTTP使用C/S架构

    • 服务器：

      • 保持开机状态
      • 固定的IP地址
      • 可以采用具有拓展性的数据中心

    • 客户端：

      • 主动与服务器进行通信
      • 可以间断性连接
      • 可以采用动态IP地址
      • 客户端进程之间相互不同通信

  • peer-to-peer (P2P)架构

    平等，有利于文件的分享，对等节点可以从其他对等节点请求服务，同时可以为其他对等节点提供服务，对等节点可以间歇性互连或者改变IP地址，导致管理非常复杂。

    1. 服务器不需要总是处于开机状态
    2. 同一个主机既可以是客户端也可以是服务器
    3. IP地址不固定
    4. 任意终端系统之间可以直接通信

2.1.2可供应用程序使用的运输服务

也就是一个应用程序需要运输层提供的运输服务有哪些

  • data integrity （数据完整性）：有些应用要求数据100%可靠传输，有些不需要
  • timing （时间保证）：有些应用要求低时延
  • throughput （吞吐量保证）：有些应用要求有最小吞吐量保证才有效
  • Security （安全保证）：加密，数据完整性

2.2 Web应用的应用层协议——HTTP 80

在学习Web应用的应用层协议HTTP之前，我们需要了解一些web应用的基本术语。

  • 一个网页可以包括多个对象（HTML file ，JPEG image，Java applet，audio file等），但都包括一个基本的HTML文件，它可以包括多个引用对象，每个对象通过一个URL来寻址。
  • URL:统一资源定位系统（uniform resource locator)，如http://www.w3school.com.cn/media/media_mi.asp
    • http 表示使用的是http协议
    • www 服务器名
    • W3school.com.cn 域名（关于域名，会在2.4DNS中详细讲到）
    • www.w3school.com.cn 这整个是主机名
    • /media/media_mi.asp 后面的都是路径名

2.2.2 HTTP基本概况

  • 是Web应用的应用层协议，使用 C/S架构
  • 使用TCP进行可靠的传送，是无状态的
    1. 客户端首先向服务器进程发起TCP连接请求（通过创建socket）端口号80
    2. 服务器进程接受来自客户端进程的TCP连接请求
    3. HTTP消息（应用层协议消息）在浏览器（HTTP客户端）和web服务器（HTTP服务器）之间交换
    4. TCP连接关闭

2.2.2 两种HTTP连接

我们已经知道了HTTP使用C/S架构，那么C和S之间如何建立连接传输数据呢，消耗时间的情况是怎样的呢？

基本概念：

RTT：time for a small packet to travel from client to server and back. 即数据报来回C/S一趟的时间

  1. 非连续HTTP连接
    • C/S通过TCP连接最多发送一个对象，然后关闭TCP连接。即服务器发送响应之后就会关闭连接
    • 下载多个对象需要建立多个TCP连接
    • 响应时间 = 2RTT + file transmission time
      • 一个RTT用于建立TCP连接
      • 一个RTT用于HTTP请求

  1. 持续HTTP连接（默认）

    • C/S建立一个TCP连接，在客户端和服务器之间多个对象可以基于一个TCP连接发送,但必须是同一个主机（服务器）

      如www和MAIL就是两个不同的服务器，就不能使用同一个TCP连接

    • 和非持续HTTP一样，先用一个RTT建立连接，之后进行数据传输。但持续HTTP连接中，服务器响应一次之后不会关闭连接，所以客户机还可以继续发送请求，基于此分为流水线式请求和非流水线式请求：

      • 流水线式请求（默认）：客户机可以一个接一个连续发生请求，就是说在前一个请求接收到响应之前就可以产生新的请求，服务器可以一个接一个连续发送响应对象。因此节省RTT时延，可能所有引用对象只花费一个。
      • 非流水线式请求：客户机只能在前一个响应接收到之后才能发出新的请求，也就是为每一个引用对象的请求和接收都使用一个RTT时延。

我们已知有这两种HTTP连接，很明显效率最高的就是流水线式的持续HTTP连接。在实际情况中，默认使用的都是流水线式的持久连接。

2.2.3 HTTP报文格式

  1. 请求Request信息格式——客户机给服务器的！

    HTTP请求报文由请求行（request line）、请求头部（header）、空行和请求数据4个部分组成。

话不多说，直接上图！

  • 请求行：

    • 请求方法：包括GET，POST，HEAD
      • GET：请求指定的页面信息
      • POST：提交表单（添加信息），数据包含在请求体中
      • HEAD：类似于GET，不过返回的响应中没有具体内容，只用于获取报头，请求返回对象的响应报文首部
      • 请求URL：路径名
      • HTTP协议及版本：有HTTP/1.0和HTTP/1.1，基本上是后者（放在请求行的最后）

  • 请求头部/报文头部

    请求头部由关键字/值对组成，每行一对，下面举几个常见的：

    1. User-Agent : 产生请求的浏览器类型，也就是当前用户的浏览器信息

对标HTTP，我们还需认识一个协议——FTP。简单说HTTP是面向网页的，而FTP是面向文件的。

​ FTP是File Transfer Protocol, 文件传输协议；

  1. FTP协议：用于使用TCP在FTP服务器和网络中的客户端计算机之间上传和下载文件（在两主机间传输控制信息），可以将文件从一个主机复制到另一个主机。FTP使用TCP的服务在客户端和服务器之间传输文件，会建立两个连接，一个用于TCP端口号20上的数据传输，另一个用于TCP端口号21上的控制信息（命令和响应）。就是说需要建立两个链接：控制连接（21 传输用户认证信息，浏览目录，文件请求命令）和 数据连接（20 专门用于传输文件数据）。

  • HTTP：无状态，控制信息是带内传送，请求和响应都是在传输文件的TCP连接中发送

怎么理解有状态呢？我们传输文件的时候可以断点续传，就是说服务器会维护一个文件的传输的状态，已经发送的数据。无状态就是指服务端对于客户端每次发送的请求都认为它是一个新的请求，上一次会话和下一次会话没有联系

3. 运输层

3.1 概述

运输层的主要协议有TCP和UDP，我们已知的大多数应用层协议都是基于TCP的，只有少数情况会使用UDP，比如DNS、流媒体（流式多媒体）、DHCP、SNMP。但注意DNS是同时使用TCP和UDP协议，两者都会使用。DHCP动态主机配置协议，DHCP和DNS都是应用层协议。TCP和UDP协议都提供全双工服务。

TCP ：

  • 可以提供的服务：数据的可靠传输、流量控制、拥塞控制
  • 不能提供的服务：时间控制（时间保证）、最小吞吐量保证（带宽保证）、安全性
  • 是面向连接的方式(有连接的建立和拆除过程），提供端到端服务，提供全双工服务
  • 全双工（Full Duplex）是通讯传输的一个术语。通信允许数据在两个方向上同时传输，它在能力上相当于两个单工通信方式的结合。全双工指可以同时（瞬时）进行信号的双向传输（A→B且B→A）。指A→B的同时B→A，是瞬时同步的。
  • 半双工，指可以分时进行信号的双向传输（A→B或B→A）。指A→B时，不能B→A；B→A时，不能A→B。即数据发送和数据接受是分时进行的。（汽车的单边放行。）
  • 单工，只允许A→B传送信息，而不能B→A传送 。（汽车的单行道。）

UDP ：

  • 可以提供的服务：数据的不可靠传输
  • 不能提供的服务：可靠性、流量控制、 拥塞控制、时间保证、最小吞吐量保证、安全性（就是TCP不能的加上TCP能的）
  • 是无连接的方式(没有连接的建立 和拆除过程)，提供全双工服务

3.2 多路复用/多路分解

  1. 为什么需要多路复用/多路分解呢？

​ 我们知道，在网络上主机与主机之间的通信，实质上是主机上运行的应用进程之间的通信。例如，当你通过Http上网浏览网页时，实质上是你所访问的主机的服务器进程与你本机的浏览器进程在进行通信。试想一下，当你在上网的同时，还挂着QQ，还使用ftp下载大文件，这时就有三个网络上的进程与你的主机上的三个进程进行通信，那么系统是怎么样正确地把接收到的数据定位到指定的进程中的呢？也就是说，系统是怎么把从ftp服务器发送过来的数据交付到ftp客户端，而不把这些数据交付到你的QQ上的呢？反过来考虑，系统又是如何精确地把来自各个应用进程的数据发到网络上指定上的主机（服务器）上的对应进程的呢？这就是多路分解与多路复用的作用了。

为了更好地理解多路复用和多路分解，我们还需要用到我们操作系统的知识：

操作系统用于帮我们管理计算机可以使用的资源，当我们的程序要使用一个资源的时候，可以向操作系统申请，再由操作系统为我们的程序分配和管理资源。通常当我们要访问一个内核设备或文件时，程序可以调用系统函数，系统就会为我们打开设备或文件，然后返回一个文件描述符fd（或称为ID，是一个整数），我们要访问该设备或文件，只能通过该文件描述符。可以认为该编号对应着打开的文件或设备。

​ 同理，当我们的程序要使用网络时，要使用到对应的操作系统内核的操作和网卡设备，所以我们可以向操作系统申请，然后系统会为我们创建一个套接字Socket，并返回这个Socket的ID，以后我们的程序要使用网络资源，只要向这个Socket的编号ID操作即可。而我们的每一个网络通信的进程至少对应着一个Socket。向Socket的ID中写数据，相当于向网络发送数据，向Socket中读数据，相当于接收数据，而且这些套接字都有唯一标识符——端口号。

​

  1. 有了上面的基础后，再来说说什么是多路分解和多路复用。

​ 每个运输层的报文段中设置了几个字段，包括源端口号和目的端口号等。

  • 多路分解就是，在接收端，运输层检查这些字段并标识出接收套接字，然后将该报文定向到该套接字。其工作方式可以简单地认为是这样的，主机上的每个套接字有被分配一个端口号，当报文到达主机时，运输层检查报文段中的目的端口号，并将其定向到相应的套接字。

  • 多路复用就是从源主机的不同套接字中收集数据块，并为每个数据块封装上首部信息从而生成报文段，然后将报文段传递到网络层中去。

  1. 在运输层中，针对TCP和UDP，分别有无连接的多路复用多路分解和有连接的多路复用和多路分解：

    1. 无连接的多路复用和多路分解

      在运输层，无连接的网络传输是通过UDP来实现的。UDP报文中只有源端口号和目的端口号，一个UDP套接字是由一个含有目的IP地址和目的端口号的二元组来全面标识的。这里没有过多地说明IP地址，是因为IP地址是网络层的知识，所以现在还没有提及，我们现在只须知道，IP地址对应着一台主机，而端口号对应着一台主机上的一个进程（或套接字）。

    2. 面向连接的多路复用和多路分解

      我们可以知道，网络上主机间的进程间通信，实质上是通过套接字来实现的。在运输层中面向连接的网络传输多使用TCP，而TCP套接字和UDP套接字之间有一个细微的差别，就是，TCP套接字是由一个四元组（源IP地址、源端口号，目的IP地址，目的端口号）来标识的。这样，当一个TCP报文段从网络到达一台主机时，主机会使用全部4个值来将报文段定向，即多路分解到相应的套接字。

3.3 无连接传输协议UDP

UDP是一个无连接的协议，提供“尽力而为”的服务，随时可能会丢包、交付失序等问题。那为什么还需要UDP协议呢？

  • UDP的优点

    • 无连接创建，减少时延
    • 简单：在发送方，接收方无连接状态
    • 段首部小
    • 无拥塞控制，能尽快地传输（会丢包）

  • UDP报文段格式

    首部字段包括：源端口，目的端口，报文段长度，检查和（都是16字节）

  • UDP校验和

    • 发送方
      1. 将段内容处理为16比特整数序列
      2. 计算检查和：段内容相加，如果有进位要回卷加到后面去，最后变反码
      3. 发送方将检查和放入检查和字段
    • 接收方
      1. 计算接收的段的检查和
      2. 核对是否等于检查和字段的值
      3. 相等—可能有差错，不相等—有差错

3.4 可靠数据传输的原则

在学习面向连接的传输TCP之前，我们先了解如何才能进行可靠的数据传输，应该遵守什么原则？

基本概念：

  • ACK：Acknowledge character 确认发来的数据已经接收
  • NAK：Negative Acknowledae
  • rdt：reliable data transmission 可靠数据传输

3.4.1 停-等机制

摸索过程如下：

  1. 经可靠数据传输——rdt1.0
  2. 经有比特差错的信道传输——rdt2.0
  3. 处理受损的ACK/NAK——rdt2.1
  4. 具有比特差错和丢包的信道——rdt3.0（停-等协议）

最终得到停-等协议，具有以下可靠传输机制：

  • 差错检查
  • 正确确认
  • 重传机制
  • 定时器机制

停等协议的控制方法：

  • 发送方：使用0，1两个序号对分组进行循环编号，发送一个帧，并且设立定时器
    接收方：接收到帧后返回该帧的ack，若收到错误的分组则返回上一个分组的确认
    发送方：接收到ack后才发送下一个帧，或者超时重传

对于重复的报文段，或者重复的Ack确认包都进行丢弃处理。

虽然停-等协议能工作，但性能不太好，但很明显，一个一个传也太慢了，故由此进一步提出流水线协议

3.4.2 流水线机制：GBN和SR

​ 流水线协议就是指发送方允许发送多个“传输中的”、还没有应答的报文段。因此在停-等协议的基础上必须做出一点改变

  • 序号的范围必须增加

    在停-等协议中因为是一个一个传，所以序号只需要0，1就够了。但对于流水线协议而言，因为有多个报文段在传输，所以序号的范围也需要增加以便表示报文段。

  • 发送方或接收方设有缓冲区（细节“或”，兄弟萌。因为GBN无接收方缓冲区）

​ 流水线协议有两种形式，回退N帧法（go -back-N），选择性重传（Selective Repeat）。因为GBN还是有不必要的重传问题，因此提出SR，只重传没有收到ACK的分组。下面我们来详细介绍一下这两种方式：

  1. GBN（累计确认机制）

窗口内的分组分为已发送还未收到 ACK 确认的分组(黄色)，可发送还未发送的分组 (蓝色)。send_base 指向已发送但未收到 ACK 确认的分组的最小序号。

发送方：

  • 维护一个窗口，不能发送超过窗口范围外序号的分组，分组首部需要 K 比特序号，2^k^=N，允许 N 个连续的没有应答分组
  • 收到接收方返回的 ACK(n)，表示接收方收到了所有比 n 小的序号的分组(累计确认机制);可能会收到重复的 ACK 确认包
  • 定时器，对最老已经发送还未收到确认 ACK 的分组(send_base)维护一个定时器（同时只对一个分组维护定时器）
  • 定时器超时机制，timeout(n)，重传所有已经发送还没有收到 ACK 确认的分组。

接收方:

  • 接收空间只有 1 个包的大小，没有接收缓冲区。如果是期望接收序号的包，就接收下来，向上层提交，如果不是期望接收序号的包，则丢弃，返回 ACK(期望的序号); 注意:没有接收缓冲区。会发送一个重复的 ACK 确认，就是说如果pkt2没收到，那么不管收到啥包，都是只发送ack1，表示只收到pkt1，现在期望收到pkt2.
  1. SR

对标GBN，SR有以下几点改进措施

  • 接收方设立缓冲区，分别确认所有正确接收的报文段，以便最后按序交付给上层。SR既有发送窗口也有接收窗口，GBN只有发送窗口。
  • 定时器对每个没有确认的分组计时，发送方只需要重传没有收到的ACK的分组
  • 在发送窗口中，需要限制已发送但尚未应答的分组的序号

维护发送缓冲区，有个滑动窗口，包括已发送未收到 ACK 确认的分组(黄色)，已

发送已收到 ACK 确认的分组(绿色)，未发送分组(蓝色);

发送方:

  • 收到 ACK(n)，在 SR 中，ACK 确认是针对单个分组的，收到对某个分组的 ACK 确认，说明这个分组被接收方收到。
  • 定时器，对当前已发送未收到确认的所有分组各自维护一个定时器，多个定时器;
  • 定时器超时，某个序号的分组的定时器超时，则只重传超时对应的这个分组。

接收方：

  • 收到分组是期望序号的分组，则返回 ACK(n)给发送方;
  • 对于收到的分组如果不是期望序号的分组，则缓存这个分组，发送按序到达的分组的确认 ACK 给发送方。

因为发送方和接收方都有窗口，所以序号有可能对应错误，就是说这一轮发出去的的序号，接收方可能会误认为是下一轮中的数据报。所以需要限制窗口长度的大小——窗口长度小于等于序号空间的一半。这样的话，在一个序号空间中至少有两轮窗口中的值，接收方就不会认错了。

eg. 极端情况下，发送窗口的值就等于序号空间的值。发送窗口发出的包，接收方全部收到，但是返回的确认消息全部丢失，这时候，发送方会重发这一轮的0号包，但接收方就会把它当做下一轮的0号包进行接收。

  • 差错检查
  • 正确确认
  • 重传机制
  • 定时器机制
  • 流水线

3.5.2 TCP传输机制

TCP采用了一种GBN和SR的混合机制（因为虽然SR传输效率更高，但实现起来更复杂）：

  • 累计确认（GBN）
  • 单个计时器（GBN）
  • 将失序达到的报文段缓存（SR）
  • 每次重传只发送一个报文段（不论是定时器超时还是收到3个重复ACK）

可以理解为是在GBN的基础上，在接收方也使用缓冲区，

下面针对这四个机制，我们依次来认识TCP中的可靠数据传输：

  1. 累计确认机制

    TCP只确认数据流中至第一个为收到字节为止的字节。就是说确认号ACK为N，表示N之前的字节都全部收到了。

    • 正确确认机制（ACK）
      • TCP中只有正确确认
      • 接收方接收segement，通过校验和判断是否损坏，如果损坏，不发生动作，也不通知重传
      • 对收到的segement进行累计ACK确认

  2. 单个计时器

  3. 将失序的报文段缓存

  4. 每次重传只发送一个报文段（定时器超时 & 快速重传）

    什么情况下启动重传：定时器超时或者连续收到3个冗余的ACK确认

    • 定时器机制

      • 启动：每当发送一个报文且这个报文之最早的发送且未确认的，则启动定时器

      • 终止：收到确认且这个确认时最早的发送且未确认的报文段的确认，终止定时器。并将发送窗口前移，启动新的定时器。

      • 超时时间间隔：

        EstimatedRTT = (1- α)EstimatedRTT + αAsmpleRTT (typical value : α = 0.125)

        DevRTT = (1-β)*DevRTT + β * |SampleRTT - EstimatedRTT| (typical value : β = 0.25)

        TimeoutInterval = EstimatedRTT + 4*DevRTT(偏差)

    • 快速重传机制：收到3个相同的冗余的ACK确认，则认为对应的数据奥丢失，立即重传（因为定时器超时时间一般比较长，所以用这个方法提高效率）

      （注意快速重传机制中，指的是冗余的ACK确认，第一个ACK不是冗余的，所以直接点看，一般是收到第四个ACK确认才算是收到3个冗余的ACK确认）

3.5.4 TCP流量控制

流量控制：发送方不能发送太多、太快的数据让接收方缓冲区溢出。应用进程可能从接收缓冲区读取数据缓慢，因此发送速率需要匹配接收方应用程序的提取速率。——流量控制是用来防止接收方缓冲区溢出。

工作原理：

  • 在TCP连接的接收方有一个接收缓冲区，其中有一部分是接收窗口，一部分数据缓冲区
  • 接收方再报文daunt接收窗口字段中通告其接收缓冲区的剩余空间
  • 发送方要限制未确认的数据（也就是在正在发送的数据）不超过接收方窗口值，以此保证缓冲区不溢出

3.5.5 TCP拥塞控制

在介绍具体内容之前，先对基本概念和特殊名词做一些了解：

名词 意义
MSS (协商的)1个最大报文段的长度，Maximum Segment Size
ssthresh 慢启动阈值
dupACKcount 重复ACK计数标记
timeout 计数器超时
cwnd 拥塞窗口 Congestion Window

TCP中的状态主要包括3个，分别是：慢启动(slow start)、拥塞避免(congestion avoidance)、快速恢复(fast recovery)。根据状态转移图，启动时首先是慢启动状态（参考虚线箭头）：

从图中可以得出一些特殊信息：

  • 只要是timeout，都会转换到慢启动状态
  • 每次变化都应该ssthresh先变化，cwnd再变化
  • 其他两个状态收到重复的3个ACK就会进入快速恢复状态
  • 状态发生变化时，ssthresh = cwnd/2，进入拥塞避免状态时不需要改变（因为cwnd>ssthresh时就转换状态了）
  • cwnd三种变化：
    • 慢启动状态：指数增长
    • 拥塞避免状态：线性增长（1个RTT增加一个）
    • 快速恢复状态

接下来详细介绍这三种状态：

  1. 在慢启动状态：初始时，cwnd为1个报文段(MSS)，ssthresh(慢启动阈值)为64KB，dupACKcount(重复ACK计数标记为0)；

    慢启动状态下cwnd的增加规则为：每收到一个新的ACK，cwnd=cwnd+1MSS，即拥塞窗口增加1个最大报文段长度；也就是cwnd呈指数性变化。

例如第1个RTT，cwnd为1个MSS，假设接收方立即返回1个ACK，则第2个RTT开始之前，cwnd变为2个MSS，以此内推；第2个RTT，发送方发出2个MSS，接收方收到并返回2个ACK，则第3个RTT开始之间，cwnd变为4个MSS。

当cwnd大于等于慢启动阈值(ssthresh)时，状态就从“慢启动”转为“拥塞避免”状态；

  1. 在拥塞避免状态，在拥塞避免状态，cwnd的增加规则为：每收到一个新的ACK，cwnd = cwnd + MSSMSS/cwnd；注意这里不是增加1个最大报文段长度，而是增加（MSS/cwnd）个MSS，*通俗点理解就是每个RTT只增加1个MSS**。

在拥塞避免状态（或慢启动状态），当收到3个重复ACK，进入快速恢复状态，在状态转移的时候，执行的操作是将慢启动阈值设置为当前拥塞控制窗口的一半，即ssthresh = cwnd/2; 将新的拥塞控制窗口设置为新的慢启动阈值加3个MSS，cwnd = ssthresh + 3MSS；重新传输丢失的报文段

  1. 当处于快速恢复状态时比较复杂，有两种不同的操作
    • 每收到1个重复的ACK，拥塞控制窗口的变化：cwnd = cwnd + 1MSS（注意这里是每收到一个重复ACK，就增加1个MSS）；然后传输允许的新的报文段【感觉不是很合理，但是RFC原文是这样写的】
    • 当收到1个新的ACK时，从快速恢复状态返回拥塞避免状态，cwnd = ssthresh，这里的慢启动阈值是进入快速恢复状态时的计算值，也就是不发生变化

慢启动状态 快速恢复状态 拥塞避免状态

在状态中的变化 指数变化 每收1冗余ACK，+1/进入拥塞避免 线性变化

ssthresh的变化始终是ssthresh = cwnd/2，进入拥塞避免时不需要改变

3.5.6 TCP连接管理

  1. TCP 的建立:三次握手
  • 客户机向服务器发送 TCP SYN 报文段【表明准备好发送数据，不知服务器】
    • 指定初始序号(随机产生)seq=x，
    • 没有数据
  • 服务器收到 SYN 报文段, 用 SYN ACK 报文段回复。【已知客户端准备好发送数据，表明自己准备好发送(接收)数据】
    • 服务器为该连接分配缓冲区和变量（注意这里，为SYN泛洪攻击提供契机）
    • 指定服务器初始序号(初始化自己的序号 seq=y)
  • 客户机接收到 SYN ACK，用 ACK 报文段回复，可能包含数据【告诉服务器已经收到服务器确认(知道服务器准备好)，发送数据】

（SYN泛洪攻击就是发送大量的TCP SYN报文段，但最后第三步握手不去确认，服务器不断为这些半开连接分配资源，导致服务器的连接资源被耗尽）

为什么最后由客户端应答服务器确认消息的步骤不能缺少? 我们说客户端和服务器在发送数据之前，需要完成的工作包括初始化序号，以及创建缓冲区，双方需要知道对方的序号，且确认序号是正确的(即客户端要知道服务器的序号，且确保服务器知道自己的正确序号【通过 SYNACK 可以确认】;服务器也需要知道客户端的序号，且确保客户端知道自己的正确序号【通过第三次握手的 ACK 消息告知服务器】)。因此不能省略第三次握手，否则服务器不能确保客户端知道自己正确的序号。

  1. TCP的释放：四次挥手（客户机和服务器的任意一端都可以发送消息主动关闭连接）

    • 客户机向服务器发送TCP FIN控制报文段【告知服务器，客户端不再发送数据， 但是可以接收来自服务器的数据】

4.1 网络层概述

  1. 网络层和我们刚学的运输层有什么区别呢？
  • 运输层实现“进程-进程”之间的通信服务，在网络边缘的端系统中实现
  • 网络层实现“主机-主机”之间的通信服务，在端系统以及网络核心的路由器中实现
  • 主机实现全部五个层次，路由器实现底部三个层次（所以可以看到，运输层不在路由器中实现，网络层在）
  • 因特网的网络层不建立连接，路由器上不需要维护状态

  1. 因特网的网络层提供的服务（因特网是基于数据报网络，在发送数据前不需要在网络层建立连接）

    • 单一服务，即尽力而为服务(best-effort service) 。
    • 分组间的定时不能被保证;
    • 分组的接收顺序与发送顺序不一定相同;
    • 传送的分组不能保证最终交付，即网络可能未向目的地交付分组。

  2. 网络服务模型有数据报网络（因特网）和虚电路网络（ATM），数据报网络提供网络层的无连接服务，虚电路网路提供网络层的连接服务。数据报网络直接发送分组，每个分组都是独立的数据转发单元，可以自由选择转发路径，没有拥塞控制和流量控制。虚电路网络在分组传输之前需要建立源到目的计算机的数据通路-虚电路，分组沿着确定的通路进行传输，有拥塞控制和流量控制，但路由器负载增加，一旦链路出现故障，整个链路都将失效，可拓展性差。

    讲到面向连接服务，这里与运输层又有些区别。

    传输层的面向连接服务是在网络边缘的端系统中实现

    网络层的面向连接服务是在端系统及网络核心的路由器中实现

  1. 网络层实现主机到主机的通信服务。网络层的核心硬件就是路由器，根据路由器的核心功能——路由和转发，网络层能够被分为两个部分：数据平面和控制平面。
  • 数据平面的功能，即网络层中每台路由器的功能。该数据平面功能决定到达路由器输入链路之一的数据报如何转发到该路由器的输出链路之一。——转发
  • 控制平面功能，即网络范围的逻辑。该控制平面功能控制数据报沿着从源主机到目的主机的端到端路径中路由器之间的路由方式。——路由

所以在网络层的讲解中，分成两个平面来讲，这里我们对网络层的数据平面进行学习。

4.2.1 输入端口的处理和转发

输入端口的结构如下图：

  1. 线路端接模块：将一条物理链路 端接 到路由器的物理层（实现路由器的物理层功能）

  2. 数据链路处理模块：实现路由器的数据链路层功能

  3. 查找与转发模块：实现查找与转发（基于目的地转发）功能，以便分组通过路由器交换结构转发到适当的输出端口。通过查找转发表实现，转发表存储在输入端口的内存中。具体如何创建转发表这就是路由的事（路由的活和我转发有什么关系）。

    1. 如何查表：最长匹配规则

    2. 分布式交换

      • 路由处理器计算转发表，给每个输入端口存放一份转发表拷贝
      • 在每个输入端口本地做出交换决策，无需激活中央选路处理器
      • 可以避免在路由器中某个单点产生转发处理瓶颈
      • 目的：以线速完成输入端口的处理
      • 排队：如果数据报到达输入端口的速度快于输入端口将数据报转发到交换结构的速度，就会排队

4.2.2 交换结构

将内存从输入端口缓存交换（转发）到恰当的输出端口缓存中。有三种类型的交换结构：经内存交换，经总线交换，经互联网络交换

  1. 经内存交换 memory

    • 交换由CPU（路由处理器）控制完成
    • 输入输出端口类似I/O设备，分组到达输入端口—发出中断—将分组拷贝到处理器内存中—处理器查表得出适当的输出端口—分组拷贝到输出端口的缓存中（OS的知识），显而易见，需要经过总线，故交换速度受总线带宽的速度限制（每个分组穿过两次总线）

  2. 经总线交换 bus

    • 通过一条共享总线将分组直接送到输出端口，不需要路由处理器干预
    • 只有一条总线，每次只能一个分组通过总线传送
    • 路由器交换带宽受总线速率限制

  3. 经互联网络交换 crossbar

    • 到达输入段扩的分组沿水平总线穿行，直至与所希望的输出端口的垂直总线交叉点，若阻塞，则必须在输入端口排队
    • 网状的，速率高于经总线的交换结构

4.2.3 输出端口的处理

输出端口结构和输入端口类似：

取出存放在输出端口内存中的分组，并将其传输到输出链路上。当交换结构将分组交付到输出端口超过输出链路速率时，就会出现排队，这就需要输出端口的排队和缓存管理功能。如果缓冲区溢出，就会出现延时和丢包。这里重点介绍调度算法（当然这些在OS和ETOS中都有涉及，所以也很简单）

分组调度算法：

  • FIFO：先来先传输呗
  • 优先级：高优先级的分组先进性传输
  • RR：基于优先级，高优先级和低优先级轮流传输
  • 三种策略的实例：加权公平队列WFQ

4.3.2 IPv4编址

基本概念：

  • IPv4地址：分配给主机或路由器接口的32bit标识符
  • 接口：主机/路由器与物理链路之间的边界（主机和路由器都可以有多个接口，每个接口有一个IP地址）
  • 子网：为了确定网络区域，分开主机和路由器的每个接口，从而产生了若干个分离的网络岛，接口端连接了这些独立网络的端点。这些独立的网络岛叫做子网(subnet)。（就是接口之间就是一个子网）

接下来介绍IPv4的编址方式，实际上就是传统IPv4地址和CIDR编址方法：

  1. 传统的五类地址ABCDE（0/10/110/1110/11110) （要能判断一个IP是哪类IP）

    • IP地址的前五位（高五位）用于标识IP地址的类别
    • ABCDE类地址的前几位分别是0/10/110/1110/11110，A、B、C类地址为基本的IP地址
    • IP地址由网络号和主机号组成

    缺点也很明显，一个A类地址可以分配的主机有2^24^个，一个家庭或者一个主机往往用不完，一个C类又不够用，所以这种传统的编址方式逐渐被CIDR技术替代。

    在介绍CIDR之前，还有些常用特殊IP地址介绍一下

  2. 常用特殊IP地址

    • 127.0.0.1-127.255.255.254

      主要用来识别主机本身的地址，也叫做localhost，一般用来测试使用，做开发的人比较熟悉

    • 10.x.x.x,127.16.x.x - 127.31.x.x,192.168.x.x

      主要使我们私有的内网地址，也就是我们平时企业或者家里局域网所使用的地址段

    • 0.0.0.0

      主要用来标识不清楚的网络和主机的，系统遇到无法识别的网络或主机的时候会统一的归纳到这个地址

    • 255.255.255.255

      这个地址是受限的广播地址，主要指一个网段内的所有主机

  3. CIDR：无类别路由选择地址（Classless InterDomain Routing）

    • 地址中的网络部分可以任意长

    • 地址格式：a.b.c.d/x 这里x表示网络位，剩下的就是主机位

      （但其实现在在计算机中看到的只写做a.b.c.d，但这时一般会标注子网掩码）

    • 子网掩码：将IP地址中的 网络位全1，主机位全0 就是子网掩码

    • 子网号/网络号：IP地址与掩码进行按位的与运算（就是把主机位都设为0）

    属于同一个子网的两台主机可以直接发送分组进行通信，不需要路由器；但不属于同一个子网的两台主机通信，发送方需要先将分组发送到它的网关路由器，由网关路由器帮助转发分组到接收主机。至于网关路由器如何建立转发表，如何转发就是我们第五章要讲的内容了。

4.3.3 分配IP-DHCP

​ DHCP(Dynamic Host Configuration Protocol),动态主机配置协议，是一个应用层协议。当我们将客户主机ip地址设置为动态获取方式时，DHCP服务器就会根据DHCP协议给客户端分配IP，使得客户机能够利用这个IP上网。

​ DHCP客户-服务器交互场景

DHCP的实现分为4步，分别是：
第一步：Client端在局域网内发起一个DHCP Discover包，目的是想发现能够给它提供IP的DHCP Server。
第二步：可用的DHCP Server接收到Discover包之后，通过发送DHCP Offer包给予Client端应答，意在告诉Client端它可以提供IP地址。
第三步：Client端接收到Offer包之后，发送DHCP Request包请求分配IP。
第四步：DHCP Server发送ACK数据包，确认信息。

当然DHCP返回的不止是IP地址，还会给客户端第一跳路由器的自制，DNS服务器的IP地址和域名，网络掩码（用于划分网络位和主机位）

4.3.4 IP转换-NAT

​ 自IPv4出现到二十世纪90年代初，网络终端数量爆炸式增长，网络专家们意识到，这样下去，IPv4地址很快就要耗光了。于是，人们开始考虑IPv4的替代方案，同时采取一系列的措施来减缓IPv4地址的消耗。正是在这样一个背景之下，本期的主角闪亮登场，它就是网络地址转换——NAT。

​ NAT是一项神奇的技术，说它神奇在于它的出现几乎使IPv4起死回生。在IPv4已经被认为行将结束历史使命之后近20年时间里，人们几乎忘了IPv4的地址空间即将耗尽这样一个事实——在新技术日新月异的时代，20年可算一段漫长的历史。更不用说，在NAT产生以后，网络终端的数量呈加速上升趋势，对IP地址的需求剧烈增加。此足见NAT技术之成功，影响之深远。（好了不吹了，进入正题）

​ NAT（Network Address Translation）协议：在私有地址和全局地址之间转换的协议。

​ 首先什么是私有地址？私有地址是不能用在Internet上(路由器将丢弃寻址这种地址的包)的内部地址。这些地址是不能够在公网上面用的，只能用在局域网的内部。私有地址有三种：①10.0.0.010.255.255.255/8 ②172.16.0.0172.31.255.255/12 ③192.168.0.0~192.168.255.255/16 这些IP地址是用于私有的网络。与之对应的是全局地址，就是正规的自己电脑的地址，全网络承认。比如说，每个人都有自己的大名，走到哪里都能被承认，这就是自己的全局地址；但是在班级里面的外号，就是私有地址，只有班级里面的人知道，在外面别人都不知道这个外号对应的是谁。

​ 那么每个人都有一个全局地址不就挺好的了吗？为什么还要麻烦取一个“小名”内部地址呢？这就涉及到上面说过的IPv4的网络不够用的问题了。IPv4地址随着用户的增多压力不断增大，但是每一个路由器的IP地址下面都有很多的私有地址，外部消息只需要找到这个路由器，这个路由器把消息找到真正目的主机传递给它即可。每一个路由器都可以分配很多私有地址，并且不同路由器的私有地址可以重复，就如同一班的“二狗子”和二班的“二狗子”并不是同一个人一样，通过这种地址转换，能够大大增加地址的容量。

​ 除此以外，NAT协议还有为了网络安全性的考虑。内部网络有自己安全性的考虑，这样只要保证路由器不转发内部地址的消息，那么这些消息就不会轻易跑到公网上面去，从而保证安全。

​ NAT工作过程如下：

  • 外出的分组：替换每个外出分组的源IP和源端口（私有地址），远程客户/服务器把这个替换之后的全局地址、端口号作为目的IP和端口来响应。
  • 进来的分组：替换每个进来分组的目的IP和目的端口

5.2 路由算法

5.2.1 基本概念

  • 基础术语：

    • 默认路由器：与主机直接相连的第一台路由器，又叫第一跳路由器

    • 从源主机到目的主机的选路归结于从源路由器到目的路由器的选路

    • 用节点图表示网络的结构，图G = （N，E）表示N个结点和E条边的集合，c(x,y)表示节点x和节点y之间边的费用，用一个结点序列来表示路径，如（x,y,b,m,w）

  • 路由算法分类：

    • 根据算法是全局性还是分布式来划分：
      • 全局算法：用完整的、全局性的网络信息来计算最低费用路径，所有路由器拥有完整的拓扑和链路费用信息
      • 分布式算法：以迭代的、分布式的方式计算最低费用路径，节点只知道与其直接相连的链路的路由器信息和链路的费用信息，不需拥有所有网络链路费用的完整信息；通过迭代计算，并与相邻节点交换信息得出最低费用路径。
    • 根据算法是静态的还是动态划分：
      • 静态算法：路由确定后基本不变化，除非人工干预
      • 动态算法：当网络的流量负载或拓扑发生变化时，改变路径。不过容易受到选路循环、路由振荡之类问题的影响。

因特网常用的两种方法LS和DV，也就是我们接下来要介绍的路由选择算法：

  • 链路状态选路算法LS—Dijkstra算法：动态，迭代的，全局
  • 分布式距离向量算法DV：动态，迭代的，分布式

5.2.2 链路状态选路算法LS—Dijkstra算法

  • 符号定义

    • D(v)表示从源节点u到节点v的当前费用
    • P(v)表示从源节点u到节点v的路径上的前一节点（注意都是源节点）
    • N：节点集合

  • 算法：

    3. 重复步骤2直到所有节点都加入集合N

5.2.3 距离向量算法—DV

DV是一种迭代的、分布式、动态、异步选路算法。（异步表示不要求所有节点相互之间步伐一致地操作）

  • 符号定义
    • dx (y):节点x到节点y的最低费用路径的费用
    • 用Bellman-Ford方程表示dx(y) = minv{c(x, v)+ dv(y)} (就是在x,y之间插入其他节点）
  • 算法
    • 每个节点不断向邻居节点发送自己的距离向量估计值Dx
    • 当x从邻居节点收到新的距离向量估计值Dv，它使用B-F方程更新自己到其他节点的距离向量Dx

算法不难，直接上例题

  • 缺陷：会出现选路环路和计数到无穷的问题。

    当某条链接的费用减少时，我们称之为有一个“好消息”。在网络中，好消息的传递往往很迅速。

    例如，存在这样一个网络：

    ​ 某一时刻，Y检测到它到X的链路费用由4减少为1，好消息当然要告诉大家了，于是它更新了自己的距离向量，并通知了Z。Z在收到Y的更新报文后，也更新了自己的距离向量（由5减为2），并向邻居们发送更新报文。而后，Y又收到了Z的更新报文，但它发现并没有改变自己的最低费用，于是保持不变。这样，仅仅经过了两次迭代网络就达到了静止。好消息通过网络得到了迅速传播。

    但是，当链路费用增加（甚至断开）时，就不会这么简单了。

    我们看下面这个例子：

    ​ 还是X、Y、Z三个节点。此时Y检测到它到X的路径费用由4增加到了60。此时节点Z的距离向量为：d(X) = 5, d(Y) = 1, d(Z) = 0。于是Y在更新向量时发现，咦，Z到X的距离只有5诶，那可以先到Z再到X，于是Y的距离向量更新为：d(x) = 5 + 1 = 6, d(Y) = 0, d(z) = 1。我们可以发现，这个逻辑显然是错误的，因为Z到X的距离为5的前提是要经过Y，但Y更新后的路径又要经过Z，这就形成了一个选路环路（routing-loop）问题。因为Y的距离向量更新了（虽然是错误的），但它还是向Z发送了更新报文。Z收到更新报文后，比较了下邻居们到X的距离，发现经过Y的路径距离为1 + 6 = 7，小于直接到X的距离，于是Z也更新的自己的距离向量，然后又将更新后的距离向量发给Y。Y收到后又更新向量为8，然后再发给Z。。。这样循环往复，更新报文在Y和Z之间传来传去，直到第44次迭代后，Z算出它经由Y的路径费用大于50为止。此时，Z最终确定到X的最短路径费用是直接到达X的费用50，而Y也得到了最短路径是经Z到X的费用51。

    ​ 可以看出，虽然最后还是得到了正确的信息（最后的50和51是正确的！），但坏消息的传播与好消息相比实在是慢太多了！而且，如果X和Y之间的费用为10000，Z和X的费用是9999时，就会出现计数到无穷（count-to-infinity）的问题！

    ​

  • 解决办法：毒性逆转方法

    上述的选路环路问题可以通过毒性逆转的技术加以避免。它的基本思想是：如果Z的最短路径要通过邻居Y，那么它将告诉Y自己到目的节点的距离是∞。这样，Z向Y撒了一个善意的谎言，使得只要Z经过Y选路到X，它就会一直持续讲述这个谎言，这样Y也就永远不会尝试从Z选路到X了，也就避免了环路问题。（可见，有时善意的谎言也不是件坏事，也是为了你好啊T^T）

    ​ 我们将毒性逆转技术应用于上例。Y在更新自己的距离向量时，发现Z到X的距离是∞，于是它将d(x)无奈地更新为60，并向Z发送了更新报文。Z收到报文后更新自己的d(X)为50（直接选路到X），并发给Y更新报文（此时因为Z不需要经过Y进行选路，因此将告诉Y自己到X的距离为50）。Y在接收到Z的报文后，重新将距离更新为1 + 50 = 51，并告诉Z自己到X的距离是∞（实际是51）。Z收到报文后，发现最低耗费并没有改变，因此算法进入静止状态。

5.2.4 LS与DV的异同

最大的区别：

链路状态选路算法LS：是全局式的，每个节点与所有其他节点广播交流，得到邻居节点到网络中其他节点的最低费用，并告知邻居节点自己当前到其他节点的最低费用。

距离向量算法DV：是分布式的，每个节点只与邻居互相交流，只告知与其直接相连链路的费用

5.3 路由协议—OSPF，RIP，BGP

现在我们已经将两种常用路由算法掌握了，但网络层是使用协议来提供服务的，算法应该由协议使用。因此接下来介绍的就是网络层的路由协议：

  • OSPF：ISP域内路由协议，采用链路状态路由算法（全局式）
  • BGP：ISP域间路由协议，采用距离向量算法（分布式）

基本概念：

  • 自治域AS：可以将路由器聚合为若干的区域，称为自治系统或自治域
  • 网关路由器：位于AS的边缘，有链路连接到其他AS的路由器
  • intra-AS域内路由：位于相同AS内的主机和路由器之间的路由，域内所有路由器必须运行相同的域内路由协议
  • inter-AS域间路由：AS之间的路由，网关执行域间路由协议

转发表的配置由域内和域间路由算法完成

  • intra-AS 路由确定AS内部的目 的地址的转发表项
  • inter-AS & intra-AS 确定外部目的地址的转发表项

三种路由协议的区别如下表，具体情况我们接下来分节描述

5.3.1 ISP域内路由协议：OSPF和RIP

域内路由协议也称为内部网关协议(interior gateway protocols,IGP)

目前应用最广泛的域内路由协议包括:

  • RIP: Routing Information Protocol，使用距离向量算法
  • OSPF: Open Shortest Path First (IS-IS protocol essentially same as OSPF)，使用链路状态算法
  • IGRP: Interior Gateway Routing Protocol (Cisco proprietary for decades, until 2016)

5.3.2 ISP域间路由协议：BGP

BGP为每个AS提供如下功能:

  • eBGP: 从其邻居AS获得子网的可达信息，域间，在网关路由器与其他AS域的网关路由器之间建立
  • iBGP: 传播可达信息给AS内部的所有路由器，域内，在网关路由器与其所在AS域的路由器之间建立

注意，每个路由器都需要同时运行域内和域间路由协议。

5.5 SDN体系结构

在传统网络中，由路由器完成数据的转发和路由。除此之外，还有SDN体系结构，将数据平面和控制平面分离，基于流完成转发的任务。

  • SDN字面意思是软件定义网络，其试图摆脱硬件对网络架构的限制，这样便可以像升级、安装软件一样对网络进行修改，便于更多的APP（应用程序）能够快速部署到网络上。传统网络是分布式控制的架构，每台设备都包含独立的控制平面，数据平面。SDN的定义就是其字面意思，控制与转发分离，管理与控制分离只是满足SDN的一种手段。
  • OpenFlow只是满足SDN的一种协议，运行在 SDN 控制器和(SDN 控制的)交换机或其他(实现 OpenFlow API 的)设备之间。它的核心思想很简单，就是将原本完全由交换机/路由器控制的数据包转发过程，转化为由OpenFlow交换机（OpenFlow Switch）和控制服务器（Controller）分别完成的独立过程。

SDN关键特征：

  • 基于流的转发
  • 数据平面与控制平面分离
  • 网络控制功能:位于数据平面交换机外部
  • 可编程的网络

6. 链路层和局域网

6.1 链路层概述

  1. 链路层提供的服务

    1. 将数据报封装成帧，增加头部，尾部信息（在数据帧头部使用MAC地址来标识元目的MAC地址，不同于IP地址）

    2. 在相邻节点之间可靠传输数据帧，之前也学习了在运输层如何实现数据的可靠运输

      为什么在协议栈的两层都提供这种可靠运输服务呢？两者有什么异同呢？

      ​ 链路层的可靠传输服务，主要用于易产生高差错率的链路，其目的在于本地纠正一个错误，而不是像在运输层或应用层那样进行端到端的数据重传来进行纠正。

      ​ 而运输层提供可靠传输服务，是因为并不是所有的链路层都提供了可靠数据传输服务，要实现端到端的可靠数据传输，还需要运输层自己提供可靠传输服务。

      ​ 两者都是通过确认和重传实现的。

    3. 流量控制：控制发送节点向直接相连的接收机诶单发生数据帧的频率

    4. 差错检查：可能由信号衰减，噪声引入等产生，接收方通知重传或丢弃数据

    5. 错误纠正：接收方标识和纠正比特错误，不需要重传

    6. 半双工和全双工：半双工模式下，链路的两个节点都可以发送数据，但不能同时发送

  2. 链路层实现的位置

    还是那三句话：主机实现全部五个层次，路由器实现底部三个层次，交换机实现底部两个层次。

    在主机上，链路层的主体部分是在网络适配器上实现的（也就是我们俗称的网卡）

在本章我们不仅会学到链路层提供的服务包括差错检查、多路访问、链路层寻址（ARP），还会讲到链路层的技术如以太网和虚拟局域网等。

对于链路层提供的差错检查和纠错服务，在之前讲运输层TCP/UDP头部检查和的时候提了一嘴，链路层的检查机制有一定相似性，但更多的是他们之间的差异。

6.2 差错检查和纠错技术

​ 当然差错检查和纠错技术不能保证接收方检测到所有的比特差错，只需要选择一个合适的差错检测方案使未检测到的情况发生的概率很小即可。差错检查和纠错技术效果越好，越复杂，开销也越大（理所当然）

​ 三种主要的差错检查技术：

  • 奇偶校验：最基本的方法
  • 检查和：常用于运输层
  • 循环冗余检测：常用于链路层

6.2.1 奇偶校验

  1. 一比特奇偶校验

    • 原理

      • 在要发送的信息D（d位）后面附加一个奇偶校验位，校验位表示信息D中1的个数是奇数（奇校验）还是偶数（偶校验），共发送d+1位
      • 接收方检测收到的信息（d+1位）中1的个数，如果不符，则至少有一个比特出现差错

    • 特点

      • 只能检测，无法纠正，可以查出任意奇数个错误，但偶数个错误就不能发现了，
      • 如果比特差错概率小，差错独立发生，就可以用这个方法
      • 如果差错集中突发，一帧中未检查出差错的概率达到50%
  1. 二比特奇偶校验
    • 原理
      • 将要传信息D(d比特)划分为 i 行 j 列( i 个组，每组 j 位)
      • 对每行和每列分别计算奇偶值
      • 结果的 i+j+1 个奇偶比特构成了帧的差错检测比特。
    • 特点
      • 可以检测并纠正单个比特差错(数据或校验位中)
      • 能够检测(但不能纠正)分组中任意两个比特的差错。

6.2.2 检查和

  • 原理
    • 发送方：将数据每两个字节16bit的，全部求和，注意回卷（就是将最高位的进位加回结果的尾部），最后取反得到检查和，放到首部一起发送
    • 接收方：对接收的信息（包括检查和）按相同的方式求和（这里就体现了之前为什么要取反，取反之后，这里加起来看是否由差错一目了然）
      • 全1，收到的数据无差错
      • 其中有0，收到的数据出现差错
  • 特点
    • 分组开销小:检查和位数比较少
    • 差错检测能力弱
    • 适用于运输层(差错检测用软件实现，检查和方法简单、快速)。

可以发现检查和这种方法的差错检查能力是很弱的，远远无法满足链路层的要求（链路层提供可靠数据帧传输服务），链路层的差错检测由适配器中专用的硬件实现，采用更强的CRC方法。

6.2.3 循环冗余检测

  • 概念

    • CRC编码/多项式编码 R ：r位
    • 接收方生成的多项式 G ：r+1位 ，发送方和接收方共同选定的，最高有效为（最左边）是1
    • CRC编码的位数 r : 就是接收方生成的多项式G的位数-1，也是R的位数
    • 发送的数据 D

  • 原理

    • 发送方:

      • 计算出一个r位附加比特R，添加到D的后面产生DR(d+r 比特)
      • DR能被生成多项式G模2运算整除（就是说用G除D x 2^r^ ,余数值刚好是R ），一起发送。

      模2运算：加法不进位，减法不借位

      乘以2^r^ ：就是比特模式左移r，也就是在后面加r个0

      反过来也能算出CRC编码R的值：将数据D后面添加r个0，除以给定的生成多项式G，所得余数即为R(r位)

    • 接收方:

      • 用生成多项式G去除接收到的DR(d+r比特)
        • 余数非0 ：传输发生差错
        • 余数为0: 传输正确，去掉尾部r位，得所需数据D

    总得来说就是，在数据D后面加上一个CRC编码R，用选定的多项式G去除DR，看结果为0则无差错

  • 特点

    • 能检测小于r+1 位的突发错误、任何奇数个错误

6.3 多路访问协议

链路层有两种网络链路：

  • 点对点链路:链路两端各一个节点。一个发送和一个接收。如，点对点协议PPP
  • 广播链路:多个节点连接到一个共享的广播信道。任何一个节点传输一帧时，信号在信道上广播，其他节点都可以收到 一个拷贝。

点对点没啥好看，比较简单，我们来学习在广播链路的情况中如何实现数据的传播，因为大家都在用，所以需要信道共享技术，来协调多个发送和接收点对共享广播信道的访问，相关就是多路访问协议。

  • 多路访问协议
    • 目的：协调多个节点在广播信道上的传输，如果出现冲突（就是两个以上的节点同时传输），所有冲突的帧都会受损丢失
    • 分类：
      • 信道划分协议：将信道划分成片/时隙，给节点分配专用的片/时隙（具有公平性，效率都不高）
      • 随机访问协议：不划分信道，允许冲突，但可以“恢复”（不具有公平性）
      • 轮流协议： 通过轮流访问避免冲突，但节点越多，轮流时间越长（具有公平性）

6.3.1 信道划分协议

包括TDMA，FDMA，CDMA（码分多路，不要求）

  • 时分多路接入TDMA（Time Division Multiple Access）

    • 原理

      • 将时间划分为时间帧，每个时间帧再划分为N个时隙，分别分配给N个节点（就像一个时间帧是一轮，帧里面每个时隙分给一个节点用）

    • 特点

      • 避免冲突，公平，但节点速率不高，效率不高（因为节点必须等它的时隙到来）

  • 频分多路接入FDMA（Frequency Division Multiple Access）

    • 原理
      • 将总信道的带宽划分成N个较小的信道，再分配给N个节点（就是同时传输，每个节点用一部分信道）
    • 特点
      • 避免冲突，公平，但节点带宽有限，效率不高

6.3.2 随机访问协议-ALOHA，CSMA，CSMA/CD

  • 基本思想
    • 发送节点以信道全部速率发送，发生冲突时，冲突的每个节点分别等待一个随机时间，再重发，直到帧(分组)发送成功。

  1. ALOHA协议

    1. 时隙ALOHA

      • 原理

        • 假设所有帧长L比特，时间线被划分为若干等长的时隙（长度为一帧传输的时间，保证一帧一个时隙传完）
        • 所有节点同步传输，如果一个时隙有多个节点同时传送，所有节点都能检测到冲突
        • 冲突的话，以概率p在后续的每一个时隙重传该帧，直到成功

      • 特点

        • 当只有一个活动节点时，以全速传输（和TDMA不同）
        • 分散的，每个节点检测冲突并决定何时重传
        • 有多个活动节点是效率变低

      • 效率计算

        • 时隙可以分为成功时隙，空闲时隙，冲突时隙
        • 假设有N个节点，节点在每个时隙以概率p传输，则一个节点成功传送的概率是p(1-p)^N-1^
        • N个节点中任意一个节点成功的概率Np(1-p)^N-1^ ,取极限 = 1/e = 0.37
        • 同理计算空闲时隙和冲突时隙，37%时隙空闲，37%时隙成功，26%时隙有冲突

  1. 纯ALOHA

    • 原理
      • 相对于时隙ALOHA来说，就是非时隙，完全分散的协议。
      • 节点有数据帧发送，就立即传输
      • 如果有冲突，在冲突帧传完后，以概率p立即重传；或者以概率（1-p）等待一个帧传完，再以概率p重传
    • 效率计算
      • 取极限为1/2e = 0.18,仅仅为时隙ALOHA协议的一半

    在学习了这两种ALOHA协议我们可以发现，ALOHA协议的特点有：

  • 每个节点的传输与广播信道上其他节点的活动是相互独立的
  • 一个节点开始传输时并不知道其他节点正在传输（是相互独立的，当然不知道了）
  • 发送冲突时不会停止传输（而是等待，以一定概率重传）
  • 效率不高

接下来我们要学习CSMA就是针对ALOHA的缺陷进行改进的协议

  1. 载波侦听多路访问协议CSMA

    • 原理

      • 具有载波侦听功能：某个节点在发送之前，先监听信道，则回退一段时间，再监听信道

      回退时间：以太网的二进制指数回退（在CSMA/CD上也是这种方法）

      当传输一个给定帧时，在该帧经历了一连串的n次碰撞后，结点随机地从{0,1,2…. 2^n-1}中选择一个K值

      回退时间就是 = K * （网卡发送512bit数据的时间）

      Eg. 假设是100Mbps的以太网，那么发送512bit = 512/(100 x 10 ^6^) = 5.12*10 ^-6^ 秒，在第三次碰撞后，节点选择K = 4的概率有多大？在该以太网上对应多少秒的时延？

      ​ 第一次碰撞:{0,1}第二次碰撞:{0,1,2,3}第三次碰撞:{0,1,2,3,4,5,6,7}

      所以第三次碰撞取到4的概率为1/8，在100Mbps的以太网上对应的时延是 4 * 5.12*10 ^-6^ 秒

  • 特点

    • 发前监听，可减少冲突
    • 但由于传播时延的存在，仍有可能出现冲突，并造成信道浪费（监听的时候别的数据帧没来，传播一段时间后别的帧来了）

    发生冲突时，节点仍继续传输，但该帧已经被破坏，是无用的数据，这就导致信道传输时间被浪费

  1. 带冲突检测载波监听多路访问协议CSMA/CD

    • 原理
      • 具有“载波侦听”和“冲突检测”两个规则
      • 冲突检测：在发送同时进行冲突检测，一旦检测到冲突就立即停止传输，尽快重发（和CSMA的不同之处，为了缩短无效传送时间，提高信道的利用率）。中止传输后，会回退等待一个随机时间，重新从侦听信道开始。
    • 详细运行机制
      • 适配器从网络层获得一个数据报，封装成帧，准备发送
      • 如果适配器侦听到信道空闲，开始传输帧;如果检测到信道繁忙，将等待一段时间，直到侦听到信道空闲，开始传输帧
      • 在传输过程中，适配器会同时监听是否有其他适配器的信号能量如果适配器在整个帧的传输过程中，没有监听到其他信号，则完成该帧的传输;如果监听到来自其他适配器的信号，则中止传输帧
      • 中止传输后，适配器会等待一个随机时间，重新执行步骤2

6.3.3 轮流协议

有两种方式：

  • 来自中心站的轮询：主节点“邀请”从节点依次传送
  • 令牌传递：控制令牌顺序从一个节点到下一个节点

6.4 交换局域网

6.4.1 链路层寻址和ARP

每个节点都有它的网络层地址和链路层地址，网络层地址我们已经在网络层中学过了。网络层地址就是节点在网络中分配的一个唯一IP地址，用于把分组送到目的IP网络，Ipv4长度为32bit，接下来我们就来看看链路层地址有什么区别，寻址需要什么协议

链路层地址：MAC地址或物理地址、局域网地址

  • 概念/特点：
    • 用于把数据帧从同一个节点传送到另一个节点（同一子网，不同子网之间的事交给网络层干）
    • 节点“网卡”本身所带的地址（唯一），长度通常为6字节，48比特
    • 用16进制表示，1字节表示为两个16进制数
    • 网卡的MAC地址是永久的,生产时就固化在它的ROM里面了
  • MAC地址分配
    • 前三个字节由专门机构IEEE管理，后三个字节厂家自行分配
    • MAC地址是平面结构，网卡在任何网络中都有同样的MAC地址
    • IP地址具有层次结构，当节点移动到不同网络时，节点的IP地址发生改变
    • 特殊地址：MAC广播地址FF-FF-FF-FF-FF-FF，由全1组成（启用ARP协议时会用到）

广播信道的局域网中，一个节点发送的帧，在信道上广播传输，其他节点都可能收到这个帧。不过一般情况下，一个节点只向一个特定的结点发送。

局域网（LAN）中每个网卡都有唯一的局域网地址，网卡也被称为适配器。由网卡来负责MAC地址的封装和识别。

​ 发送适配器：将目的MAC地址封装到帧中，并发送，所有其他适配器都会收到这个帧

​ 接收适配器：检查帧的目的MAC地址是否与自己的MAC地址相匹配，如果匹配则接收该帧，取出数据报；如果不匹配则丢弃。

学到这，我们已经学习了节点的三种不同地址表示：应用层的主机名，网络层的IP地址，链路层的MAC地址；在不同地址间转换，从主机名到IP地址我们使用DNS域名系统，那么从IP地址到MAC地址，就是由我们待会要学习的ARP地址解析协议实现。就像DNS协议属于应用层协议，由HTTP等应用层协议使用一样，ARP也是属于网络层协议，而不是属于链路层协议。

  • DNS域名系统:将主机名解析到IP地址。DNS为在因特网中任何地方的主机解析主机名。 应用层协议。
  • ARP地址解析协议:将IP地址解析到MAC地址。ARP只为在同一个LAN上的节点解析IP地址。网络层协议。

ARP缓存表表：局域网上的每个节点（主机、路由器）都有这个表，用于进行IP/MAC地址映射，数据样式<IP address ;MAC address ; TTL>，TTL表示存活时间，超时地址映射将被删除，一般为20分钟。

  • 那么什么时候需要使用ARP协议呢？

    当一个网络接口（主机或路由器）要发送数据帧，需要知道发送链路另一端的接收端口的MAC地址，首先查找自己的ARP缓存表，

    • 如果缓存表中对应的IP-MAC地址队存在，则直接发送数据帧
    • 如果没找到，启动ARP协议

    ARP协议内容：源主机发送一个ARP请求报文，封装在一个广播数据帧中，同一个子网的主机/路由器都能收到这个广播帧，只有目的IP地址一致的主机/路由器会返回一个相应报文给查询主机，返回的数据帧中包含目的主机（它自己）的MAC地址，源主机就在它的ARP缓存表中缓存这个地址对。

可以发现，对于同一个局域网的两台主机来说，还是很简单的，如果缓存表中有，就直接发数据帧；如果缓存表中没找到，那就发个广播帧，也能很快找到目的主机的MAC地址。那么对于不同局域网的两台主机如何实现通信呢？

主机A经路由器R发送数据报给主机B，假设主机A知道主机B的IP地址，假设主机A知道网关路由器R的IP地址(通过DHCP协议) ，假设主机A知道网关路由器R的MAC地址(通过ARP协议)

  1. 主机A构建IP数据报，源地址是A的IP地址，目的地址是B的IP地址
  2. 主机A构建链路层数据帧，目的MAC地址是路由器左边端口的MAC地址
  3. 数据帧从主机A发送到路由器R
  4. 路由器R收到数据帧，抽取出数据报递交到IP层（需要重新封装一下，现在的数据帧上的MAC地址是不对的，现在封装改成B的MAC地址）
  5. 路由器R将该数据报封装成链路层帧，目的MAC地址为主机B的MAC地址

可以得出结论：IP地址在传递过程中始终不变! MAC地址需要根据实际链路的发送和接收端口进行设置!

  • 总结一下ARP协议的特点：
    • ARP缓存表是动态更新的
    • ARP缓存表只针对相同局域网的网络接口查找MAC地址

6.4.2 以太网

在前面我们提到了很多次以太网，那么什么是以太网呢？以太网Ethernet就是现在一种最为著名的有线局域网技术。

  1. 以太网成功的原因：是第一个广泛使用的局域网技术，简单又便宜，版本在不断更新，数据速率更高、成本更低
  2. 以太网物理拓扑结构包括：总线形和星形（目前流行的）
  3. 以太网帧结构

发送适配器将IP数据报封装成以太网帧，传递到物理层；接收适配器从物理层收到该帧，取出数据报传递给网络层。

  • 前同步码：8字节，用于发送方和接收方的时钟同步（接收方一旦收到连续的8字节前同步码，就知道有帧来了），无效信号，收到就删掉
  • 目的MAC地址和源MAC地址：都是6字节
  • 类型字段：表示使用不同的网络层协议
  • 数据：数据报字段。以太网的最大传输单元MTU就是1500字节，最小是46字节，不够要填充到够才能发送
  • CRC：4字节，检测数据帧中是否出现比特差错
  1. 以太网向网络层提供不可靠的无连接服务（都是对标TCP的）
    • 无连接服务：发送方适配器和接收方适配器之间不需要“握手”
    • 不可靠服务：接收到的帧可能有比特错误（因为收到正确帧不发确认帧，收到出错帧直接丢也不发否定帧，发送适配器不重发出错帧）
  2. 以太网的MAC协议：使用CSMA/CD协议（无时隙，无公平性）

6.4.3 链路层交换机

  • 链路层交换机的工作原理(交换表的建立是通过自学习)

    • 存储转发数据帧
    • 检查达到的数据帧的MAC地址，有选择的转发数据帧到一个或多个输出链路，当数据帧被转发到一个共享网段时，使用 CSMA/CD 来访问共享链路

  • 交换机完成过滤和转发的功能，通过交换表完成

    • 过滤：判断一个帧是应该转发还是丢弃

    • 转发：决定一个帧应该被指向哪个接口，并引导到该接口

    • 原理

      • 当交换机收到数据帧:

        1. 记录到达链路和发送主机的MAC地址

        2. 使用数据帧的目的MAC地址，在转发表中检索

        3. 如果在转发表条目中找到对应的MAC地址

        4. 执行{

          ​ 如果目的MAC地址对应的端口与数据帧的达到端口相同则丢弃该数据帧

          否则，转发该数据帧到条目指定的端口}
          1. 否则，向除到达端口之外的所有端口转发(flood)
  • 和路由器的比较
都是存储转发设备 都有转发表
路由器 网络层设备，检查网络层头部 使用路由算法得到，基于IP地址转发 第三层的分组交换机
交换机 链路层设备，检查链路层头部 通过泛洪、自学习得到，基于MAC地址转发 第二层的分组交换机
  • 对广播帧的处理
    • 集线器：收到数据帧，会将其向自己的所有端口转发; 不能隔离冲突域和广播域;
    • 交换机：收到广播帧，会将其向自己的所有端口转发; 能够隔离冲突域，不能隔离广播域;
    • 路由器：收到广播帧，会根据目的IP地址项对应端口转发; 能隔离冲突域，能隔离广播域

6.4.4 虚拟局域网

VLAN是一种将局域网设备从逻辑上划分(注意，不是从物理上划分)成一个个网段(或者说是更小的局域网LAN)，从而实现虚拟工作组(单元)的数据交换技术。 VLAN这一新兴技术主要应用于交换机和路由器中，但目前主流应用还是在交换机之中。不过不是所有交换机都具有此功能，只有三层以上交换机才具有此功能。

  • VLAN 的划分方法:
    • 按照端口号划分
    • 按照 MAC 地址划分
    • 按照网络层地址划分(IPv4)
  • VLAN的功能/好处：
    • 端口的分隔。即便在同一个交换机上，处于不同VLAN的端口也是不能通信的。这样一个物理的交换机可以当作多个逻辑的交换机使用。（VLAN可以跨越多个交换机）
    • 流量隔离，网络的安全。不同VLAN不能直接通信，杜绝了广播信息的不安全性。
    • 灵活的管理。更改用户所属的网络不必换端口和连线，只更改软件配置就可以了。
